## 二、实验内容：两种路径对比

### 路径 A：纯提示词驱动

**思路**：一次性或分阶段输入长提示词，让模型生成完整的 `train.py` 和 `app.py`。

**Prompt 示例：**

> "请帮我写两个 Python 脚本。第一个 `train.py`：用卷积神经网络（CNN）在 MNIST 数据集上训练，保存模型为 `mnist_model.pth`。第二个 `app.py`：用 Streamlit 做前端，让用户在画布上手写数字，调用模型做实时预测展示。"

**关键点**：记录模型是否一次就跑通，有没有出现路径引用错误或库缺失的问题。


---

### 路径 B：Agent + Skills 驱动

**思路**：利用模型的 Agent 能力（比如 Cline 的自动执行权限）配合 Skills（能力集）来完成。

**关于 Skill**：Skill 是一组预定义的函数或工具集。比如 **Frontend Skill** 可能包含自动生成 Streamlit 绘图组件的模板，**DNN Skill** 则包含标准化的模型评估代码。

**执行流程：**

1. 给 Agent 下任务："启动项目，用你的神经网络 Skill 搭建训练流水线，再结合前端 Skill 完成交互界面。"
2. **观察 Agent 的行为**——它会先创建目录、装依赖、写代码、跑测试，甚至根据报错自动修复。
3. **记录**：截图 Agent 自动 Debug 的过程，这是报告中体现"Agent 优势"的关键证据。

## 三、两种方法的优缺点对比

下表可以直接用在报告和 PPT 里：

|维度 |纯提示词方式 |Agent + Skills 模式 |
|---|---|---|
|**生成方式** |静态生成，依赖人工复制运行 |动态执行，自主创建文件并调试 |
|**纠错能力** |需要人工反馈报错，重新生成 |能自己捕捉报错并自我修正 |
|**模块化** |倾向于生成大段单体代码 |根据 Skills 拆分为模块化组件 |
|**效率** |初期快，后期调试成本高 |初期配置稍繁琐，但端到端完成率极高 |
|**适用场景** |简单、一次性执行的任务 |复杂、需要和环境交互的工程 |

## 四、实验总结与心得

结合之前的项目经验（比如医疗资源分析或农作物模型）来写总结：

- **自动化跨越**：强调从"模型生成代码"到"模型操作环境"的转变。
- **知识沉淀**：Skills 本质上是一种"工程化经验"的封装，用好它可以大幅降低重复开发前端 UI 的门槛。
- **反思**：Agent 虽然强，但提示词的精确度（Prompt Engineering）仍然是决定 Agent 能不能高效调用 Skill 的关键。
